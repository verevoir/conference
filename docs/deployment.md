# Deploying to GCP

The conference app runs on Cloud Run backed by AlloyDB (Postgres), GCS (asset storage), and Google OAuth. All infrastructure is managed via Terraform in `infra/`.

## Prerequisites

- [gcloud CLI](https://cloud.google.com/sdk/docs/install) installed and authenticated
- [Terraform](https://developer.hashicorp.com/terraform/install) >= 1.5
- [Docker](https://docs.docker.com/get-docker/) installed
- A GCP project with billing enabled

## 1. Authenticate

If you have multiple GCP accounts, create a named gcloud configuration to avoid conflicts:

```bash
gcloud config configurations create verevoir
gcloud config set project verevoir
gcloud auth login
gcloud auth application-default login
gcloud config configurations activate default  # switch back to your default
```

Then activate it per-terminal (or use [direnv](https://direnv.net/) with an `.envrc`):

```bash
export CLOUDSDK_ACTIVE_CONFIG_NAME=verevoir
```

## 2. Configure Terraform

```bash
cd infra
cp terraform.tfvars.example terraform.tfvars
```

Edit `terraform.tfvars` with your values:

| Variable | Description |
|----------|-------------|
| `project_id` | GCP project ID |
| `region` | GCP region (default: `europe-west2`) |
| `conference_name` | Resource naming prefix |
| `gcs_bucket_name` | GCS bucket name for assets |
| `google_client_id` | Google OAuth client ID |
| `seed_admin_id` | Your Google `sub` claim prefixed with `google-` (see below) |
| `docker_image` | Artifact Registry image URL |
| `domain` | Custom domain for the app |

**Finding your seed admin ID:**

```bash
gcloud auth print-identity-token \
  | cut -d. -f2 \
  | base64 -D \
  | python3 -c "import sys,json; print('google-' + json.load(sys.stdin)['sub'])"
```

## 3. Deploy (three stages)

Deployment runs in three stages because Cloud Run needs a container image to exist before it can create the service.

### Stage 1 — Enable APIs and create Artifact Registry

```bash
cd infra
terraform init
terraform apply \
  -target=google_project_service.apis \
  -target=google_artifact_registry_repository.app
```

This enables all required GCP APIs and creates the Docker registry. API enablement can take a minute or two to propagate — if later stages fail with "API not enabled", wait and retry.

### Stage 2 — Build and push the Docker image

```bash
cd ..  # back to conference project root

# Authenticate Docker with Artifact Registry (one-time)
gcloud auth configure-docker europe-west2-docker.pkg.dev

# Build for amd64 (Cloud Run target) and push
docker build --platform linux/amd64 -t europe-west2-docker.pkg.dev/verevoir/verevoir-conference/app:latest .
docker push europe-west2-docker.pkg.dev/verevoir/verevoir-conference/app:latest
```

### Stage 3 — Deploy everything else

```bash
cd infra
terraform apply
```

This creates:
- **VPC + subnet** for private networking
- **AlloyDB** cluster and primary instance (password auto-generated, stored in Secret Manager)
- **GCS bucket** for asset storage
- **Cloud Run** service reading DATABASE_URL from Secret Manager
- **HTTPS load balancer** with Google-managed SSL certificate
- **IAM** bindings for the Cloud Run service account

## 4. DNS

After `terraform apply` completes, grab the load balancer IP:

```bash
terraform output lb_ip_address
```

Create an A record in your DNS provider:

```
A  conference.lab.verevoir.io  →  <lb_ip_address>
```

The Google-managed SSL certificate will provision automatically once DNS propagates (can take up to 20 minutes).

## Security

- **AlloyDB password** — auto-generated by Terraform (`random_password`), stored in Secret Manager. No human ever sees or manages it.
- **DATABASE_URL** — injected into Cloud Run from Secret Manager at runtime, not stored in env vars or tfvars.
- **terraform.tfvars** — gitignored. Contains OAuth client ID and seed admin ID but no secrets.
- **Terraform state** — contains the generated password. Use a [remote backend](https://developer.hashicorp.com/terraform/language/backend) with encryption for production.

## Updating

To deploy a new version:

```bash
# Build and push new image
docker build --platform linux/amd64 -t europe-west2-docker.pkg.dev/verevoir/verevoir-conference/app:latest .
docker push europe-west2-docker.pkg.dev/verevoir/verevoir-conference/app:latest

# Trigger new Cloud Run revision
gcloud run services update verevoir-conference \
  --region europe-west2 \
  --image europe-west2-docker.pkg.dev/verevoir/verevoir-conference/app:latest
```

## Tearing down

```bash
cd infra
terraform destroy
```

Note: AlloyDB deletion is permanent. GCS bucket deletion requires `force_destroy = true` if it contains objects.
